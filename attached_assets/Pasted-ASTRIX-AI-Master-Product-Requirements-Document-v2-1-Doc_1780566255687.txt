2. The big picture (plain‑English explainer)
Product teams in SaaS companies get customer feedback everywhere: support tickets, Slack, emails, sales calls, spreadsheets, feature request boards. It is messy, incomplete, and biased.

They need to answer two questions:

What should we build next and why?

Did what we shipped actually work?

Today:

Signals are scattered.

Loud customers win, even if they are small.

Teams ship features and move on without checking if problems are solved.

ASTRIX AI is a smart notebook + calculator + scoreboard for product teams:

Collects customer messages in one place (Signals).

Connects each message to the customer and their revenue (Accounts).

Groups similar messages into bigger problems (Problems).

Scores each problem 0–100 based on business impact (Opportunities).

Records what the team decided to do and why (Decisions).

Tracks what was shipped (Launches).

Measures change at Day 7 and Day 30 (Outcome Reviews).

Forces a final verdict: Solved / Partially Solved / Not Solved / Regressed (Verdicts).

The core loop:

Signals → Problems → Opportunities → Decisions → Launches → Verdicts

This loop is sacred. Anything that doesn’t strengthen this loop is not part of MVP.

ASTRIX combines:

A retention‑worthy daily/weekly workspace for product managers.

A “Cursor for PMs” layer (AI drafts decision memos, PRD skeletons, stories).

An Accountability OS that ties every launch to evidence and a verdict.

3. Vision, mission, core promise
3.1 Long‑term vision
ASTRIX AI becomes the operating system for product decisions and post‑launch accountability in B2B SaaS companies.

In the long run:

Every meaningful product change is tied to evidence (signals/accounts), a decision, and a verdict.

Leadership sees which bets worked, which failed, and what was learned.

PMs defend priorities and outcomes with structured reasoning instead of politics.

3.2 Mission
Give every product team a single, evidence‑backed system to decide what to build, ship it, and prove whether it worked.

3.3 Core promise
Know what to build next — and whether it worked — with proof.

6. Product principles (rulebook)
Non‑negotiable principles:

Core loop is sacred: if a feature doesn’t strengthen
Signal → Problem → Opportunity → Decision → Launch → Verdict,
it is not MVP‑critical.

Evidence first: every insight and AI output traces back to signals, accounts, or metrics (citations/links).

Account impact over noise: prioritization is based on ARR, segment, and account impact, not just raw ticket count.

User in control: AI suggests and drafts; users own final grouping, scores, decisions, and verdicts. No black‑box auto prioritization.

A launch without a verdict is incomplete: system nudges and reminds until verdict is submitted.

Explainability over magic: scoring, suggestions, recommendations must be inspectable and understandable.

Fast, focused, credible: MVP must feel light for founders/PMs; no heavy enterprise complexity.

7. Habit loops & retention design
Goal: PMs and HoPs use ASTRIX every week (and often daily) because core product rituals run on it.

7.1 Core rituals
Daily / every few days — Signals Triage

Page: Signals list

Actions: review new signals, confirm AI tags, link to accounts, attach to problems.

Outcome: inbox of noise turns into structured evidence.

Weekly — Opportunity Review (Monday or start of sprint)

Page: Opportunities list + compare

Participants: HoP + PM(s), sometimes Founder.

Actions:

Filter by segment/ARR.

Review top 10 opportunities by score.

Create/adjust decisions.

Outcome: clear set of “chosen bets” for the week/sprint.

Per launch — Outcome Review Cadence

Page: Launch detail

When: baseline at pre‑launch, Day 7, Day 30.

Actions: update signal counts, ARR at risk, severity mix; write PM notes; choose final verdict.

Outcome: every launch has a verdict and proof summary.

Monthly / quarterly — Accountability & Learning Review

Page: Dashboard + Launches list

Actions:

Review solved vs not‑solved launches.

Review which segments/accounts got value.

Outcome: board/executive‑friendly view of which bets paid off.

7.2 Retention mechanisms
“Reviews due” widget on dashboard: shows pending Day 7/30 reviews and launches without verdict.

Launch reminders via email + in‑app for Day 7 and Day 30.

Weekly summary email (later): top opportunities, new signals, launches needing review.

“What changed since last visit?” summary panel on dashboard:

New signals count.

New problems or opportunities above threshold.

Launches whose status changed.

9. Core user journey (end‑to‑end)
Happy path:

User signs up (email/password or Google).

User creates a workspace (name, timezone).

User optionally loads a sample workspace to understand value.

User uploads signals CSV or adds a few manually.

User uploads accounts CSV.

System auto‑suggests signal→account matches (by domain); user confirms.

User reviews signals, edits AI‑suggested metadata (severity, sentiment, product area).

User groups signals into problems (manual + AI suggestions).

System generates and scores opportunities automatically.

User reviews ranked opportunities, selects one.

User creates a decision (Build/Fix/Experiment/Defer/Reject) with rationale.

System drafts a decision memo + PRD skeleton; user edits.

User creates a launch from the decision (owner, launch date, expected outcome, target metrics).

System reminds user at Day 7; user completes outcome review.

System reminds user at Day 30; user completes outcome review.

User submits final verdict (Solved/Partially/Not/Regressed) with rationale.

System generates a proof summary card.

Dashboard updates: top opportunities, active launches, reviews due, and new proof summary.

12. AI architecture
12.1 Approach
AI is used for classification, clustering suggestions, and drafting content — never for silent/autonomous changes. Users always approve or edit outputs.

12.2 Providers
Primary: Gemini (for free tier and light tasks).

Premium: Groq / similar for faster, higher quality generation in paid tiers.

Experimental: NVIDIA NIM and similar services may be used where free rate limits make sense for prototyping, but the system must not assume “free forever” production usage.

Provider abstraction: single internal API that routes to chosen model per workspace/plan.

12.3 AI‑powered features
Signal: title, severity, sentiment, product area.

Problems: suggested clusters.

Opportunities: explanations, recommended action.

Decisions: rationale, assumptions, risks, alternatives.

Artifacts: decision memo, PRD skeleton, user stories.

Launches: expected outcomes, target metrics, proof summaries.

12.4 Output requirements
Each AI output shows:

Suggested value.

Short reasoning where useful.

Links/citations to underlying signals/accounts/metrics.

“AI‑generated, editable” indicator.

12.5 Boundaries
User controls final grouping, scores, decisions, verdicts.

No black‑box “auto prioritize backlog for you” features.

No automatic launch creation without user action.

12.6 Quotas
Per plan, approximate AI call budgets:

Free: ~100/month.

Pro: ~1,500/month.

Business: ~5,000/month.

Enterprise: custom.

Per action: 1 call per memo/PRD/proof summary generation; classification jobs batched.

14. Backend architecture (Supabase)
14.1 Components
Auth: Supabase Auth for email/password + Google.

DB: Postgres with full schema and indices.

RLS: Row‑Level Security for workspace isolation.

Storage: for CSVs and optionally exported files.

Edge Functions:

CSV ingestion (parse → insert → enqueue AI classification).

AI wrapper calls.

Reminder scheduler (Day 7/30).

Artifact generation (for heavier models).

Billing webhooks (Dodo + Stripe).

14.2 Implementation guidance
Start with schema + auth, not AI.

Build first vertical slice: Login → Workspace → Signals list → Signal detail.

Then extend to: Problems → Opportunities → Decisions → Launches.

Don’t wire every route at once; go module by module.

15. Access control & security
Every record belongs to a workspace.

Users can read/write only records for workspaces where they’re members.

Owners manage members and billing.

Members cannot delete workspace or change billing.

Security requirements:

RLS on all workspace‑scoped tables.

No client‑side secrets.

Privileged operations only via edge functions.

Audit logging of critical actions.

HTTPS everywhere.

Secrets in Supabase env vars.

19. Risks & mitigations
Scope creep → enforce core loop gate, use explicit cut list.

Over‑automation → keep manual‑first flows, AI assist only.

RLS mistakes → test with multiple workspaces, build debug tools.

AI hallucination → ground outputs in stored data; show citations; allow overrides.

CSV edge cases → robust parsing, clear errors, admin tools.

Competitors adding similar AI features → stay focused on accountability loop (verdict + proof), which is less crowded.
