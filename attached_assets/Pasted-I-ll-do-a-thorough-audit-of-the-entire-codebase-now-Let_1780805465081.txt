5. Security & Architecture Gaps
Missing Auth Guards
src/pages/onboarding/ — ProtectedRoute wraps these in App.tsx but the WorkspaceContext doesn't validate that the user has not yet completed onboarding, so a fully onboarded user can freely revisit /onboarding/step-1 and overwrite workspace state
src/contexts/AuthContext.tsx — user is set to MOCK_USER on any sign-in; no real token validation, no JWT verification, no session expiry
Missing Role Checks
Maker vs. Viewer RBAC — usePlan.ts gates features by billing plan but the Maker/Viewer role distinction referenced in Pricing.tsx is never enforced in any app page
Admin route guard — src/components/auth/AdminRoute.tsx exists but since AuthContext returns a mock user with no real role field, the guard can be trivially bypassed by any authenticated mock user
Write actions — Signal creation, decision commits, artifact publishing — none check whether the current user's workspace role permits writes
Multi-Tenant Workspace Logic Missing
src/contexts/WorkspaceContext.tsx — activeWorkspace is set from mockDb; there is no validation that the signed-in user is actually a member of the selected workspace
src/lib/api.ts — All queries accept a workspace_id parameter but it's sourced from mock context; no server-side row-level security or workspace membership check exists
Workspace switcher in AppLayout.tsx — Allows switching to any workspace in mockDb; in production this must be scoped to the user's memberships only
Dangerous Client-Side Logic
src/lib/api.ts — Entire data layer is client-side. All "authorization" is implicit. Moving to Supabase without RLS policies will expose all rows to all authenticated users
src/pages/onboarding/Step2Data.tsx — Contains the explicit code comment: "In a real app, we would save these to the database here" — this page does nothing on submit
src/lib/supabase.ts — The dummy client will silently succeed on all calls; any component that imports supabase directly and doesn't use src/lib/api.ts will receive empty data with no error
Missing Loading / Error / Empty States
Most list pages have a basic empty illustration but filtered/searched views (Signal Explorer, Decisions History) default to blank table rather than "No results for these filters"
No global error boundary — A thrown JS error in any page component will crash the full app
No network error handling — When api.ts is replaced with real Supabase calls, rejection paths are not handled in most pages (only Contact.tsx and SignalNew.tsx have try/catch)
No optimistic update rollback — Mutations that succeed in UI but fail on server would leave stale state
